# mcu_agent

Supervises the standard **micro-ROS Agent** process (eProsima Micro-XRCE-DDS-Agent, wrapped by
`micro-ROS/micro-ROS-Agent`) that bridges a microcontroller's micro-ROS session onto normal
ROS 2 topics over USB serial. This package does **not** implement the XRCE-DDS protocol itself —
it starts/monitors/restarts the real agent binary and publishes a connection-health topic.

It is intentionally **generic and reusable**: it has no knowledge of what messages/topics the
microcontroller publishes. Once the agent is running, whatever the firmware publishes (`/imu`,
`/lidar`, `/motor_encoder`, ...) shows up as ordinary ROS 2 topics for any other node to consume.
No custom message package or per-project code is needed in `mcu_agent` itself — copy this
package into a new workspace as-is to bring up a new microcontroller.

## Architecture

```
Microcontroller (firmware)  <--XRCE-DDS over USB serial-->  micro_ros_agent  -->  ROS 2 topics
    micro-ROS client                                        (spawned & supervised
                                                               by mcu_agent_node)
```

- **Firmware** — runs on the MCU (e.g. ESP32), is a micro-ROS *client*. Not part of this package.
- **`micro_ros_agent`** — the real bridge. A separate executable, built once via the setup script
  below. Does the actual XRCE-DDS ↔ ROS 2 DDS translation.
- **`mcu_agent_node`** (this package) — owns the serial connection; starts `micro_ros_agent` as a
  subprocess, restarts it if it dies, and publishes `/mcu/agent/status` so other nodes/operators
  can see connection health. It never touches the bridged data itself.

## One-time setup

The real `micro_ros_agent` executable is not packaged for apt — it must be built from source
once, into this workspace:

```bash
bash src/mcu_agent/scripts/setup_micro_ros_agent.sh
```

This clones `micro_ros_setup` (branch matches `$ROS_DISTRO`, e.g. `jazzy`) into
`src/micro_ros_setup`, then uses it to fetch and build Micro-XRCE-DDS-Agent into `src/uros`. It
may prompt for `sudo` (system dependencies via `rosdep`) and can take several minutes. Both
`src/micro_ros_setup` and `src/uros` are gitignored — they're regenerated by the script, and must
be rebuilt separately in every new workspace (the built binary lives under that workspace's
`install/`, it does not carry over when you copy just the `mcu_agent` package elsewhere).

The script sets `EXTERNAL_SKIP` to work around a known `rosdep` scoping bug (see
[Common issues](#common-issues) below) — adjust or drop that line if the sibling packages in your
workspace differ from this one.

## Usage

### Running

```bash
ros2 launch mcu_agent mcu_agent.launch.py serial_port:=/dev/ttyACM0 baud_rate:=115200
```

Both launch arguments are optional and default to the values above.

### Parameters (`mcu_agent_node`)

| Parameter | Default | Exposed as launch arg? | Meaning |
|---|---|---|---|
| `serial_port` | `/dev/ttyACM0` | yes | Serial device the MCU is connected on. |
| `baud_rate` | `115200` | yes | Must match the firmware's configured baud rate. |
| `agent_executable` | `micro_ros_agent` | no (node default only) | Name of the agent binary to spawn. |
| `restart_delay_sec` | `2.0` | no (node default only) | Cooldown before retrying after the agent dies. |

To override `agent_executable`/`restart_delay_sec` you currently need to pass them via a
parameters file or `--ros-args -p name:=value` to `ros2 run mcu_agent mcu_agent_node` directly,
since the launch file only forwards `serial_port`/`baud_rate`.

### Checking it's healthy

```bash
ros2 topic echo /mcu/agent/status
```

`std_msgs/msg/Bool`, published at 1 Hz — `true` while `micro_ros_agent` is up, `false` while it's
down/restarting. This only reflects whether the **process** is alive, not whether the
microcontroller is actually sending valid data — see the first item below for why that
distinction matters.

### Reusing this package in a different project

1. Copy the whole `mcu_agent/` folder into the new workspace's `src/`.
2. Run `setup_micro_ros_agent.sh` again inside the *new* workspace (the built agent binary is
   workspace-local, in `install/`, and does not travel with the package source).
3. Launch as above with the new project's `serial_port`/`baud_rate`. No code changes needed
   regardless of what the new firmware publishes.
4. Carry over the domain-ID fix described below — it applies to any project using
   `micro_ros_agent`, not just this one.

## Common issues

### `ros2 topic echo` shows nothing, even though `/mcu/agent/status` is `true`

**Root cause:** `micro_ros_agent` does not read `ROS_DOMAIN_ID`. It only has a `-d/--discovery
<port>` flag, defaulting to port `7400` — i.e. **it always behaves as if it were on domain 0**,
regardless of what's in its environment. Meanwhile any normal `rclcpp`/`rclpy` node (including
whatever you use to consume the bridged topics, and `ros2 topic echo` itself) *does* honor
`ROS_DOMAIN_ID`. If your shell's default domain isn't `0` (many machines set a custom one in
`.bashrc` to avoid colliding with other robots on the same LAN), the agent and everything else end
up listening on different UDP ports and never see each other — even though the agent's own log
looks completely healthy (session established, participant/topic/publisher/datawriter all
created; that bookkeeping is just talking to the MCU over XRCE-DDS and says nothing about whether
the bridge is reachable from the ROS 2 side).

**Fix:** force `ROS_DOMAIN_ID=0` for the whole launch, ideally hard-coded into whatever script or
launch file starts this pipeline (not left to the caller's shell default):

```bash
ROS_DOMAIN_ID=0 ros2 launch mcu_agent mcu_agent.launch.py ...
```

On a dedicated robot computer, set it once system-wide in `/etc/environment` instead, so every
terminal/service on that machine agrees automatically.

### Two `micro_ros_agent` processes end up fighting over the same serial port

Only one process can meaningfully hold the serial device. If a previous launch wasn't shut down
cleanly (e.g. a child got orphaned instead of exiting with its parent), a second launch can start
a *new* agent while the old one is still silently holding the port — both read from the same
stream, corrupt each other's framing, and neither delivers clean data despite no obvious error.

**Fix:** before relaunching, make sure nothing stale is left:

```bash
ps aux | grep micro_ros_agent
fuser /dev/ttyACM0        # shows every PID with the port open — should be at most one
```

Kill anything unexpected (`kill -9 <pid>`), including orphaned `mcu_agent_node`/consumer
processes from the same stale run — mcu_agent_node's own watchdog will otherwise just respawn a
fresh agent on top of the mess.

### Agent connects fine but no data ever arrives, and this used to work

If domain and process-duplication are both ruled out, check for stale Fast-DDS shared-memory
segments left behind by repeatedly force-killing (`kill -9`) ROS 2 processes instead of letting
them shut down gracefully — `kill -9` skips DDS's own cleanup, and `/dev/shm/fastrtps_*` files can
accumulate across many kills in one debugging session:

```bash
ls /dev/shm/fastrtps_* 2>/dev/null | wc -l   # a large, growing count is a red flag
```

Safe to delete when no ROS 2 processes are currently running:

```bash
rm -f /dev/shm/fastrtps_*
```

Prefer `Ctrl+C` (SIGINT) over `kill -9` when stopping the pipeline normally, to avoid
accumulating this in the first place.

### `rosdep install` fails on an unrelated package's key (e.g. `python3-gz-msgs10`)

`create_agent_ws.sh` (called from `setup_micro_ros_agent.sh`) runs its own `rosdep` pass scoped to
the *entire* `src/`, not just the agent source it just cloned — if other packages in your
workspace (e.g. a Gazebo simulation) declare dependencies that aren't resolvable on your system,
the whole setup aborts on an error unrelated to the agent itself. Set `EXTERNAL_SKIP` (space-
separated rosdep keys) before running the script to skip them — see the existing line in
`setup_micro_ros_agent.sh` for the pattern; adjust the key list to whatever your workspace's other
packages actually declare.
