# mcu_agent

Supervises the standard **micro-ROS Agent** process (eProsima Micro-XRCE-DDS-Agent, wrapped by
`micro-ROS/micro-ROS-Agent`) that bridges the ESP32 microcontroller's micro-ROS session onto
normal ROS 2 topics over USB serial. This package does **not** implement the XRCE-DDS protocol
itself — it starts/monitors/restarts the real agent binary and publishes a connection-health
topic. Once the agent is running, the ESP32's micro-ROS publishers/subscribers show up as
ordinary ROS 2 topics; no further translation node is needed.

## One-time setup

The real `micro_ros_agent` executable is not packaged for apt — it must be built from source
once, into this workspace:

```bash
bash src/mcu_agent/scripts/setup_micro_ros_agent.sh
```

This clones `micro_ros_setup` (branch matches `$ROS_DISTRO`, e.g. `jazzy`) into
`src/micro_ros_setup`, then uses it to fetch and build Micro-XRCE-DDS-Agent into `src/uros`. It
may prompt for `sudo` (system dependencies via `rosdep`) and can take several minutes. Both
`src/micro_ros_setup` and `src/uros` are gitignored — they're regenerated by the script, not
committed.

## Running

```bash
ros2 launch mcu_agent mcu_agent.launch.py serial_port:=/dev/ttyACM0 baud_rate:=115200
```

`mcu_agent_node` publishes `/mcu/agent/status` (`std_msgs/Bool`, 1 Hz) — `true` while
`micro_ros_agent` is running, `false` while it's down/restarting.

## Topic contract (ESP32 firmware <-> ROS 2)

This is the interface `main_bot_hardware/RealRobotSystem` (in the `main_bot` package) reads and
writes. The ESP32 firmware — written separately, using `micro_ros_arduino` / `micro_ros_platformio`
— must publish/subscribe exactly these topics for the real robot to work:

| Direction | Topic | Type | Contents |
|---|---|---|---|
| ESP32 -> ROS 2 | `/mcu/joint_states` | `sensor_msgs/msg/JointState` | `name = [rear_left_wheel_joint, rear_right_wheel_joint, front_left_steer_joint, front_right_steer_joint]`; `position` (rad) meaningful for the 2 steer joints; `velocity` (rad/s) meaningful for the 2 rear wheel joints |
| ESP32 -> ROS 2 | `/imu` | `sensor_msgs/msg/Imu` | `header.frame_id = "imu_link"`; `angular_velocity` + `linear_acceleration` required (consumed by `config/practical/ekf.yaml`); `orientation` may be left unset |
| ROS 2 -> ESP32 | `/mcu/joint_commands` | `sensor_msgs/msg/JointState` | same 4 joint names; `position` = target steer angle (rad, clamped to +-0.52 by ros2_control); `velocity` = target rear wheel angular velocity (rad/s) |

Notes:
- `front_left_wheel_joint` / `front_right_wheel_joint` (the passive front wheels) are **not**
  part of this contract — there's no encoder for them on the real robot, so
  `RealRobotSystem` always reports 0 for their state.
- Only `sensor_msgs/msg/JointState` and `sensor_msgs/msg/Imu` are used — no custom message
  package is needed on either side.
- Baud rate and serial device must match what `mcu_agent_node` is launched with (defaults:
  `/dev/ttyACM0`, 115200 — matches `esp32-for-lkas/platformio.ini`'s USB-CDC configuration).
